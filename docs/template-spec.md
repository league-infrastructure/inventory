# Docker Node Application Template Specification

> This document describes the **template** — the starting point you get when you
> create a new project from this repository. Actual applications built from
> the template may diverge from the defaults described here.

---

## 1. Overview

This template provides a fully containerised, AI-first Node.js application
stack with the following components:

| Component | Technology | Location |
|-----------|-----------|----------|
| Backend API | Express (TypeScript) | `server/` |
| Frontend SPA | Vite + React (TypeScript) | `client/` |
| Database | PostgreSQL 16 (Alpine) | Docker image |
| ORM / Migrations | Prisma | `server/prisma/` |
| Reverse Proxy | Caddy (via Docker Swarm labels) | Swarm overlay |
| Orchestration | Docker Compose | `docker-compose.yml` / `docker-compose.prod.yml` |
| Secrets | SOPS + age → Docker Swarm secrets | `secrets/` |

## 3. Backend — Express + TypeScript

### 3.1 Runtime

- **Node.js 20 LTS** (Alpine base image)
- **Express 4.x** with TypeScript compiled via `tsc`
- **ts-node-dev** for local development hot-reload

### 3.2 Structure

See `server/src/` for the actual implementation. Key layout:

- `index.ts` — App bootstrap, middleware registration, server start
- `routes/` — Route handlers grouped by domain (e.g., `health.ts`, `counter.ts`)
- `middleware/` — Error handling, logging (pino)
- `services/` — Business logic, database access via Prisma client

### 3.3 API Conventions

- All API routes are prefixed with `/api`
- JSON request/response bodies
- Standard HTTP status codes
- Centralised error handler returns `{ error: string, detail?: string }`

## 4. Frontend — Vite + React + TypeScript

### 4.1 Tooling

- **Vite** for dev server and production builds
- **React** with TypeScript
- **React Router** for client-side routing
- CSS Modules or Tailwind CSS (project choice at init time)

### 4.2 Development Proxy

Vite proxies `/api` requests to the Express backend. The proxy target is
configurable via the `VITE_API_URL` environment variable (defaults to
`http://localhost:3000` for native dev; set to `http://server:3000` in
Docker). See `client/vite.config.ts`.

## 5. Database — PostgreSQL

### 5.1 Image

Use the **official `postgres:16-alpine`** image. Minimal footprint, no
unnecessary extensions.

### 5.2 Philosophy: Postgres Does It All

This template strongly favours PostgreSQL as the single data store. Before
reaching for additional services:

| Need | Postgres Solution |
|------|-------------------|
| Document / schemaless data | **JSONB** columns with GIN indexes |
| Key-value cache | JSONB table or `UNLOGGED` tables for ephemeral data |
| Auto-incrementing IDs | `SERIAL` / `BIGSERIAL` / `GENERATED ALWAYS AS IDENTITY` |
| Pub/Sub messaging | **`LISTEN` / `NOTIFY`** channels |
| Job queues | `LISTEN` / `NOTIFY` + a jobs table with `FOR UPDATE SKIP LOCKED` |
| Full-text search | `tsvector` / `tsquery` with GIN indexes |

> **Policy:** If a developer requests MongoDB, push back and demonstrate the
> JSONB equivalent. If a developer requests Redis, push back and demonstrate
> `LISTEN`/`NOTIFY` or unlogged tables. Only install MongoDB or Redis if the
> use case genuinely cannot be served by Postgres after discussion.

### 5.3 ORM — Prisma

- Schema lives at `server/prisma/schema.prisma`
- Migrations are managed by `prisma migrate dev` (development) and
  `prisma migrate deploy` (production)
- The Prisma Client is generated into `node_modules/.prisma/client`
- For JSONB fields, use `Json` type in the Prisma schema
- For Postgres-specific features not covered by Prisma (e.g., `LISTEN`/`NOTIFY`,
  partial indexes), use `prisma.$queryRaw` or `prisma.$executeRaw`

### 5.4 Common Patterns

#### Document-style Storage (JSONB)

```prisma
model Document {
  id        Int      @id @default(autoincrement())
  kind      String
  data      Json
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  @@index([kind])
}
```

```sql
-- Query a nested field
SELECT * FROM "Document" WHERE data->>'status' = 'active';

-- GIN index for fast JSONB queries
CREATE INDEX idx_document_data ON "Document" USING GIN (data);
```

#### Pub/Sub with LISTEN / NOTIFY

```typescript
import { Client } from 'pg';

const subscriber = new Client({ connectionString: process.env.DATABASE_URL });
await subscriber.connect();
await subscriber.query('LISTEN new_order');

subscriber.on('notification', (msg) => {
  const payload = JSON.parse(msg.payload!);
  console.log('New order:', payload);
});

// Publishing side
await prisma.$executeRaw`SELECT pg_notify('new_order', ${JSON.stringify(order)})`;
```

## 6. Docker Architecture

### 6.1 Three Development Environments

| Environment | How to Run | Database | Hot-Reload | Files |
|-------------|-----------|----------|------------|-------|
| **Local native** | `npm run dev` | Postgres via `docker compose up db` | Yes (native ts-node-dev + Vite) | Server & client run on host |
| **Docker dev** | `npm run dev:docker` | Postgres in Docker (`docker-compose.yml`) | Rebuild to update (or uncomment volume mounts for local Docker) | All services in Docker |
| **Production** | `docker stack deploy` | Postgres in Docker (`docker-compose.prod.yml`) | No | Swarm stack with secrets |

### 6.2 Docker Files

All Docker-related files live in the `docker/` directory:

- **`Dockerfile.server`** — Multi-stage production build (compile TS → slim runtime image)
- **`Dockerfile.server.dev`** — Development build with ts-node-dev
- **`Dockerfile.client`** — Multi-stage production build (Vite build → Caddy static server)
- **`Dockerfile.client.dev`** — Development build with Vite HMR
- **`Caddyfile.client`** — Static file serving with SPA fallback
- **`entrypoint.sh`** — Loads Docker Swarm secrets from `/run/secrets/*` as environment variables
- **`dev-server-start.sh`** — Waits for DB, runs Prisma migrations, starts dev server

### 6.3 Swarm Secrets Pattern

Docker Swarm mounts secrets as files under `/run/secrets/`. The
`docker/entrypoint.sh` script reads each file and exports it as an
uppercased environment variable:

- `/run/secrets/db_password` → `$DB_PASSWORD`
- `/run/secrets/api_key` → `$API_KEY`
- `/run/secrets/session_secret` → `$SESSION_SECRET`

### 6.4 Caddy and Domain Names

Production deployments sit behind a Caddy reverse proxy that reads Docker
Swarm labels for automatic HTTPS and routing.

- **Wildcard domain:** `*.jtlapp.net`
- **Default pattern:** `<appname>.jtlapp.net`
- Labels on the service `deploy` block drive Caddy configuration

## 7. Secrets Management

### 7.1 At-Rest Encryption — SOPS + age

Secrets are encrypted in the repository using [SOPS](https://github.com/getsops/sops)
with [age](https://github.com/FiloSottile/age) encryption, following the
[league-infrastructure wiki](https://github.com/league-infrastructure/league-infrastructure/wiki/Repository-Secrets-with-SOPS---age).

**Repository files:**

| File | Committed? | Contents |
|------|-----------|----------|
| `.sops.yaml` | Yes | Lists all public keys (developers + CI) |
| `secrets/dev.env` | Yes | Encrypted development secrets |
| `secrets/prod.env` | Yes | Encrypted production secrets |
| `secrets/*.env.example` | Yes | Plaintext templates showing required vars |
| `.env` | **No** (gitignored) | Decrypted local secrets |

**Workflow:**

```bash
# Decrypt for local development
sops -d secrets/dev.env > .env
docker compose up

# Edit secrets in-place (decrypts to editor buffer, re-encrypts on save)
sops secrets/dev.env

# Onboard new developer: add their age public key to .sops.yaml, then:
sops updatekeys secrets/dev.env
sops updatekeys secrets/prod.env
```

### 7.2 Runtime Secrets — Docker Swarm

In production, secrets are loaded into the swarm and mounted as files:

```bash
# Create swarm secrets from decrypted values
sops -d secrets/prod.env | while IFS='=' read -r key value; do
  echo "$value" | docker secret create "$(echo "$key" | tr '[:upper:]' '[:lower:]')" -
done
```

The `docker/entrypoint.sh` script converts these file-mounted secrets into
environment variables at container start.

### 7.3 Required Secrets

| Secret | Used By | Description |
|--------|---------|-------------|
| `db_password` | db, server | PostgreSQL password |
| `session_secret` | server | Express session signing key |
| `api_key` | server | Application API key (if applicable) |

Additional secrets are added per-application as needed (OAuth tokens,
third-party API keys, etc.).

## 8. Deployment

### 8.1 Local Native Development

```bash
npm run dev
# Starts: Postgres container (port 5433) + Express (port 3000) + Vite (port 5173)
# Frontend: http://localhost:5173
# Backend:  http://localhost:3000/api
```

Requires a local Docker daemon for the database container.

### 8.2 Docker Development

```bash
npm run dev:docker
# Builds and starts all three services in Docker
# Frontend: http://<docker-host>:5173
# Backend:  http://<docker-host>:3000/api
```

Migrations run automatically on server startup via `docker/dev-server-start.sh`.

### 8.3 Production — Docker Swarm

All production deployments target a Docker Swarm. Even single-node
deployments are initialised as swarms (`docker swarm init`).

```bash
# 1. Build and push images
npm run build:docker
docker tag myapp-server registry.example.com/myapp-server:v1
docker tag myapp-client registry.example.com/myapp-client:v1
docker push registry.example.com/myapp-server:v1
docker push registry.example.com/myapp-client:v1

# 2. Create/update swarm secrets (see section 7.2)

# 3. Deploy stack
TAG=v1 docker stack deploy -c docker-compose.prod.yml myapp

# 4. Run migrations
docker exec $(docker ps -q -f name=myapp_server) npx prisma migrate deploy
```

### 8.4 Rolling Updates

```bash
TAG=v2 docker stack deploy -c docker-compose.prod.yml myapp
```

Docker Swarm performs rolling updates by default — new containers start
before old ones are drained.

## 9. Testing

### 9.1 Test Layers

| Layer | Directory | Framework | What It Tests |
|-------|-----------|-----------|---------------|
| Database | `tests/db/` | Jest + `@prisma/client` | Migrations, JSONB queries, LISTEN/NOTIFY, constraints |
| Backend API | `tests/server/` | Jest + Supertest | Route handlers, middleware, service logic |
| Frontend Components | `tests/client/` | Vitest + React Testing Library | Component rendering, hooks, state |
| End-to-End | `tests/e2e/` | Playwright | Full user flows through the browser |

### 9.2 Test Database

Tests that hit the database use a separate Postgres instance (or a
separate database within the dev instance). Migrations are applied
before the test suite runs, and the database is reset between test
files using transactions or truncation.

```bash
npm test              # Show available test suites
npm run test:db       # Database layer only
npm run test:server   # Backend layer only
npm run test:client   # Frontend layer only
npm run test:e2e      # End-to-end (requires running containers)
```

### 9.3 CI Integration

Tests run in the CLASI software engineering process as part of ticket
execution (Stage 3). The test engineer agent coordinates:

1. Database tests after migration changes
2. Backend tests after API changes
3. Frontend tests after UI changes
4. E2E tests before any ticket is marked done

## 10. Integration Guidelines

### 10.1 Google Workspace

When connecting to Google Workspace APIs (Sheets, Drive, Calendar, Admin, etc.):

- **Authentication:** Use a Google Cloud **Service Account** with domain-wide
  delegation. Store the service account JSON key as a Docker secret.
- **Secret name:** `google_service_account` (the full JSON key file)
- **Library:** `googleapis` npm package
- **Scopes:** Request only the minimum scopes required for the feature
- For user-facing OAuth (not service-to-service), use OAuth 2.0 with the
  `passport-google-oauth20` strategy and store `google_client_id` and
  `google_client_secret` as swarm secrets.

### 10.2 GitHub OAuth

For GitHub login or API access on behalf of users:

- **Library:** `passport-github2` or direct OAuth 2.0 flow
- **Secrets:** `github_client_id`, `github_client_secret`
- **Callback URL:** `https://<app>.jtlapp.net/api/auth/github/callback`
- **Scopes:** Request only what you need (`read:user`, `user:email`, `repo`, etc.)

### 10.3 Pike 13 API

[Pike13](https://www.pike13.com/) is a business management platform for
fitness, dance, and activity-based businesses.

- **API Base URL:** `https://pike13.com/api/v2/` (or `https://<business>.pike13.com/api/v2/`)
- **Authentication:** OAuth 2.0 Bearer token
- **Secrets:** `pike13_client_id`, `pike13_client_secret`, `pike13_access_token`
- **Common endpoints:**
  - `GET /api/v2/desk/people` — list members/clients
  - `GET /api/v2/desk/event_occurrences` — list scheduled classes/events
  - `GET /api/v2/desk/invoices` — billing/invoices
  - `GET /api/v2/desk/visits` — attendance/check-ins
- **Rate limits:** Respect Pike 13's rate limits; implement exponential backoff

## 12. Technology Decisions Summary

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Backend runtime | Node.js 20 + Express | Mature, widely understood, excellent TypeScript support |
| Frontend framework | React + Vite | AI-friendly (largest training corpus), fast dev builds |
| Language | TypeScript everywhere | Type safety, better AI code generation, shared types |
| Database | PostgreSQL 16 (Alpine) | Single data store strategy; JSONB, LISTEN/NOTIFY reduce service count |
| ORM | Prisma | Type-safe queries, declarative schema, migration management |
| Container orchestration | Docker Compose + Swarm | Simple, no Kubernetes overhead, swarm secrets built-in |
| Reverse proxy | Caddy | Automatic HTTPS, Docker label-based config |
| Secrets at rest | SOPS + age | Modern, no GPG complexity, composable key management |
| Secrets at runtime | Docker Swarm secrets | File-mounted, no env var leakage in image layers |
| Testing | Jest + Vitest + Playwright | Each layer tested with the appropriate tool |
| AI process | CLASI | Structured requirements → architecture → tickets → implementation |

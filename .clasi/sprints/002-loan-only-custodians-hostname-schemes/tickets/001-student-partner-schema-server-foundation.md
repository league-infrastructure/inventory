# Student/Partner schema & server foundation

## Description

Add `STUDENT` and `PARTNER` to the `UserRole` enum, make `User.email` nullable,
add `User.notes`, and wire all server-side logic: migration, contracts, admin
route hardening, OAuth rejection, nullable-email audit pass, and grouped sort on
`/auth/users`.

This is the foundation that tickets 003 (transfer service) and 005 (client)
build on. Neither can be executed until this ticket is done.

## Acceptance Criteria

- [ ] `UserRole` enum in `schema.prisma` contains `STUDENT` and `PARTNER`.
- [ ] `User.email` is `String? @unique` (nullable).
- [ ] `User.notes String?` column exists on the User model.
- [ ] Prisma migration runs cleanly against the dev DB (`npx prisma migrate dev`).
- [ ] `server/src/contracts/user.ts`: `USER_ROLES` includes `'STUDENT'` and
      `'PARTNER'`; `UserRecord.email` is `string | null`; `UserRecord` gains
      `notes: string | null`.
- [ ] `POST /api/admin/users` and `PUT /api/admin/users/:id` are guarded by
      `requireAdmin` (was previously missing from this route).
- [ ] Admin create/update accept `role: 'STUDENT' | 'PARTNER'`, null/empty
      email, and optional `notes`.
- [ ] Admin create skips the email-required and email-uniqueness checks when
      role is STUDENT or PARTNER and email is null/empty.
- [ ] OAuth callback (`auth.ts`) rejects any user whose role is STUDENT or
      PARTNER — destroys session and redirects to the existing access-denied URL
      without creating a session.
- [ ] `requireAuth` middleware rejects sessions where `user.role` is STUDENT or
      PARTNER (returns 401 or 403).
- [ ] `oauth.ts:92,235` — invariant assertion added at the OAuth-authorize entry
      so `emailToClientId(user.email)` is never called with null; no `!` casts
      scattered at the call sites.
- [ ] `GET /auth/users` response includes `role` on each user object; users are
      sorted staff roles (CUSTODIAN, INSTRUCTOR, QUARTERMASTER, ADMIN)
      alphabetically first, then STUDENT and PARTNER alphabetically last.
- [ ] Nullable-email display sites guarded with `?? '—'` fallback:
      - `server/src/routes/auth.ts` — `/auth/me` response (line ~156)
      - `server/src/services/token.service.ts` (line ~72)
      - `client/src/components/AppLayout.tsx` (line ~372)
      - `client/src/pages/account/Account.tsx` (line ~112)
      - `client/src/pages/admin/UsersPanel.tsx` (line ~298)
      - `client/src/pages/admin/AdminTokens.tsx` (line ~109)
- [ ] `npx tsc --noEmit` clean in both `server/` and `client/`.
- [ ] `npm run test:server` passes (or reports only pre-existing "no test files"
      result).

## Implementation Plan

### Approach

Work bottom-up: schema → migration → contracts → routes/middleware → client
audit pass. The client audit pass (nullable email display sites) is the last
step within this ticket since it is purely defensive and does not block the
server work.

### Files to Create

- `server/prisma/migrations/<timestamp>_add_student_partner_roles/migration.sql`
  (generated by `npx prisma migrate dev --name add_student_partner_roles`)

### Files to Modify

**Schema (`server/prisma/schema.prisma`)**
- `enum UserRole`: add `STUDENT` and `PARTNER` values after `ADMIN`.
- `model User`:
  - `email String @unique` → `email String? @unique`
  - Add `notes  String?` after `avatar String?`

**Contracts (`server/src/contracts/user.ts`)**
- `USER_ROLES` array: append `'STUDENT'`, `'PARTNER'`.
- `UserRecord` interface: `email: string | null`; add `notes: string | null`.
- Any Zod input schema for user create/update: accept nullable email; accept
  `notes?: string | null`.

**Admin users route (`server/src/routes/admin/users.ts`)**
- Add `requireAdmin` as middleware on both `POST /` and `PUT /:id`.
- Accept `notes` in the request body (create and update).
- For STUDENT/PARTNER: skip the "email is required" check; skip the
  email-uniqueness check when email is null/empty (lines ~67–69, ~76).

**Auth route (`server/src/routes/auth.ts`)**
- OAuth callback: after resolving the user record by googleId, check
  `user.role`; if STUDENT or PARTNER, destroy the session and redirect
  to the access-denied URL. Do not create a session.
- `/auth/me` handler (~line 156): guard `user.email` with `?? '—'`.
- `/auth/users` handler: add `role` to the Prisma select clause; apply a
  role-order sort (define a `STAFF_ROLES` set and put non-members last),
  then alphabetical by displayName within each bucket.

**OAuth route (`server/src/routes/oauth.ts`)**
- Lines ~92 and ~235: before calling `emailToClientId(user.email)`, add an
  invariant assertion: `if (!user.email) throw new Error('OAuth user missing
  email')`. This is a programmer-error guard; STUDENT/PARTNER never reach this
  path because the OAuth callback rejects them earlier.

**requireAuth middleware (`server/src/middleware/requireAuth.ts`)**
- After the existing auth check, add: if `req.session?.user?.role === 'STUDENT'
  || req.session?.user?.role === 'PARTNER'`, respond 401 and return.

**Token service (`server/src/services/token.service.ts`)**
- Line ~72: guard `user.email` with `?? '—'`.

**Client audit — nullable email display sites**
- `client/src/components/AppLayout.tsx:372`
- `client/src/pages/account/Account.tsx:112`
- `client/src/pages/admin/UsersPanel.tsx:298`
- `client/src/pages/admin/AdminTokens.tsx:109`
  For each: replace bare `user.email` (or `currentUser.email`) with
  `user.email ?? '—'` (or equivalent null-safe rendering).

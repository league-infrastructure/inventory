# HostName scheme schema, server & MCP foundation

## Description

Add `scheme String?` to the `HostName` Prisma model and wire it all the way
through: migration, contracts, service, HTTP routes (`PUT /api/hostnames/:id`
which currently has no HTTP wiring, and `GET /api/hostnames/schemes`), and both
MCP tool schema updates (`create_hostname`, `update_hostname`).

This is the backend foundation that ticket 006 (HostName list UX) builds on.

## Acceptance Criteria

- [x] `HostName.scheme String?` exists in `schema.prisma` (nullable, no unique
      constraint, no index).
- [x] Prisma migration runs cleanly against the dev DB.
- [x] `server/src/contracts/hostName.ts`:
      - `HostNameRecord` has `scheme: string | null`.
      - `UpdateHostNameInput { name?: string; scheme?: string | null }` type
        is introduced (or existing shape is updated to include `scheme`).
- [x] `HostNameService.create()` accepts and persists `scheme`.
- [x] `HostNameService.update()` accepts and persists `scheme`; `'scheme'` is
      added to `auditFields`.
- [x] `POST /api/hostnames` accepts `scheme` in the request body.
- [x] `PUT /api/hostnames/:id` exists, is guarded by `requireQuartermaster`,
      and accepts `{ name?: string; scheme?: string | null }`.
- [x] `GET /api/hostnames/schemes` returns an array of distinct non-null scheme
      string values. Response shape: `string[]`. Chosen flat array for ticket 006.
- [x] `create_hostname` MCP tool input schema includes
      `scheme: z.string().optional()`.
- [x] `update_hostname` MCP tool input schema includes
      `scheme: z.string().optional()` alongside `name: z.string().optional()`;
      at least one of `name` or `scheme` is required (runtime guard in handler).
- [x] `update_hostname` MCP tool handler calls `hostnameService.update()` with
      `scheme` passed through.
- [x] `npx tsc --noEmit` clean in `server/`.
- [x] `npm run test:server` passes (pre-existing failures only — DB credential
      failures and one unrelated TS error; same 23 failed / 10 passed before and
      after these changes).
- [x] Manual verification: `POST /api/hostnames` with `{ name: "Knuth", scheme:
      "computer scientists" }` — `GET /api/hostnames` returns the row with
      `scheme` set. `GET /api/hostnames/schemes` returns `["computer
      scientists"]`.

## Implementation Plan

### Approach

Bottom-up: schema → migration → contracts → service → routes → MCP tools. The
MCP tools are last since they depend on the service being updated.

### Files to Create

- `server/prisma/migrations/<timestamp>_add_hostname_scheme/migration.sql`
  (generated by `npx prisma migrate dev --name add_hostname_scheme`)

### Files to Modify

**Schema (`server/prisma/schema.prisma`)**
- `model HostName`: add `scheme  String?` after `name String @unique`.

**Contracts (`server/src/contracts/hostName.ts`)**
- `HostNameRecord`: add `scheme: string | null`.
- Introduce `UpdateHostNameInput` (or update the existing inline type used by
  the service) with `name?: string; scheme?: string | null`.

**Service (`server/src/services/hostname.service.ts`)**
- `create()` method: include `scheme` in the Prisma `data` object when
  provided.
- `update()` method: include `scheme` in the Prisma `data` object when
  provided; add `'scheme'` to `auditFields` so changes are recorded in the
  audit log.

**Routes (`server/src/routes/hostnames.ts`)**
- Existing `POST /` handler: extract `scheme` from the request body and pass to
  `hostnameService.create()`.
- New `PUT /:id` handler: extract `{ name, scheme }` from request body; call
  `hostnameService.update(id, { name, scheme })`; guard with
  `requireQuartermaster`. Return the updated record.
- New `GET /schemes` handler: query
  `prisma.hostName.findMany({ select: { scheme: true }, distinct: ['scheme'],
  where: { scheme: { not: null } } })`; map results to a flat `string[]` and
  return. Register this route before `GET /:id` if an id route exists, to avoid
  the path `/schemes` being captured as an id.

**MCP tools (`server/src/mcp/tools.ts`)**
- `create_hostname` input schema (lines ~468–505): add
  `scheme: z.string().optional()`.
  In the handler, pass `scheme` to `hostnameService.create()`.
- `update_hostname` input schema: add `scheme: z.string().optional()` alongside
  existing `name: z.string().optional()`. Add `.refine(input => input.name !==
  undefined || input.scheme !== undefined, { message: 'At least one of name or
  scheme is required' })`. In the handler, pass `scheme` to
  `hostnameService.update()`.

### Testing Plan

1. `npx prisma migrate dev` — clean; `\d "HostName"` in psql shows `scheme`
   column.
2. `npx tsc --noEmit` in `server/` — clean.
3. `npm run test:server`.
4. Manual smoke tests:
   - `POST /api/hostnames { name: "Knuth", scheme: "computer scientists" }` →
     200; row has scheme.
   - `GET /api/hostnames` → includes `scheme` on all rows (null for existing,
     set for new).
   - `PUT /api/hostnames/<id> { scheme: "computer graphics terms" }` → 200;
     scheme updated.
   - `GET /api/hostnames/schemes` → returns distinct non-null schemes.
   - From an MCP client: `update_hostname({ id: <id>, scheme: "computer
     scientists" })` → persists.
   - From an MCP client: `create_hostname({ name: "Phong", scheme: "computer
     graphics terms" })` → persists.

# 004 — Manufacturer entity: Prisma model, migration, and backfill script

## Description

Manufacturer is currently a free-text `String?` on the Computer model with a
hardcoded client-side dropdown. This ticket creates the database foundation:
a new `Manufacturer` Prisma model, a migration that adds the FK on Computer
without dropping the existing string column, and a one-time backfill script
that seeds Manufacturer rows from existing string values.

The string column (`manufacturer String?`) is deliberately left in place by
this migration. It will be dropped in a follow-up migration in a later sprint
after the FK has been verified in production. See `architecture-update-r1.md`
for the two-migration rationale.

## Acceptance Criteria

- [x] A `Manufacturer` model exists in `schema.prisma` with fields: `id` (autoincrement PK), `name` (String, unique), `createdAt`, `updatedAt`, `deletedAt` (nullable — soft delete). Shape matches Category.
- [x] The `Computer` model has a new `manufacturerId Int?` field with a Prisma relation to `Manufacturer`, and a `@@index([manufacturerId])`.
- [x] The existing `manufacturer String?` field on `Computer` is still present and unchanged — no drop in this migration.
- [x] `prisma migrate dev` runs cleanly and produces a migration file under `server/prisma/migrations/`.
- [x] A backfill script exists at `server/prisma/seed-manufacturer-backfill.ts` that:
  - Selects all distinct non-null, non-empty `manufacturer` string values from Computer rows (trimmed).
  - Normalizes each to title-case before upserting (e.g. `"dell"` → `"Dell"`).
  - Upserts a `Manufacturer` row for each distinct normalized value.
  - Updates every Computer row that has a non-null, non-empty manufacturer string to set `manufacturerId` to the matching Manufacturer id.
  - Leaves `manufacturerId = NULL` on rows where `manufacturer IS NULL` or is empty/whitespace after trimming — no error thrown.
  - Is idempotent — safe to run multiple times without creating duplicates.
- [x] Running the backfill against a dev database with existing Computer rows produces correct Manufacturer rows and correct `manufacturerId` values.
- [x] The `manufacturer String?` column on Computer has a `// @deprecated — pending cleanup migration` comment in schema.prisma.

## Implementation Plan

### Approach

Modify `schema.prisma` to add the `Manufacturer` model and the `manufacturerId`
FK on Computer. Run `prisma migrate dev`. Write and test the idempotent
backfill script separately from the migration (run it manually, not as a
Prisma seed, so it can be inspected and re-run safely).

### Files to Modify

- `server/prisma/schema.prisma` — add `Manufacturer` model; add `manufacturerId` FK and index to `Computer`; add deprecation comment above `manufacturer String?`

### Files to Create

- `server/prisma/migrations/<timestamp>_add_manufacturer_entity/migration.sql` — generated by `prisma migrate dev`
- `server/prisma/seed-manufacturer-backfill.ts` — standalone TypeScript script, run with `npx ts-node prisma/seed-manufacturer-backfill.ts` from `server/`

### Schema Additions

Add after the `OperatingSystem` model and before `Computer`:

```prisma
model Manufacturer {
  id        Int       @id @default(autoincrement())
  name      String    @unique
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt
  deletedAt DateTime?

  computers Computer[]
}
```

On `Computer`, add alongside the existing `manufacturer String?`:

```prisma
  // @deprecated — pending cleanup migration (drop after manufacturerId backfill validated)
  manufacturer   String?
  manufacturerId Int?
  mfg            Manufacturer? @relation(fields: [manufacturerId], references: [id])
  ...
  @@index([manufacturerId])
```

### Migration Command

```bash
cd server && npx prisma migrate dev --name add_manufacturer_entity
```

Do NOT include `DROP COLUMN manufacturer` — additive only.

### Backfill Script Outline

```typescript
// server/prisma/seed-manufacturer-backfill.ts
import { PrismaClient } from '@prisma/client';
const prisma = new PrismaClient();

function toTitleCase(s: string): string {
  return s.trim().replace(/\b\w/g, (c) => c.toUpperCase());
}

async function main() {
  const rows = await prisma.$queryRaw<{ manufacturer: string }[]>`
    SELECT DISTINCT TRIM(manufacturer) AS manufacturer
    FROM "Computer"
    WHERE manufacturer IS NOT NULL AND TRIM(manufacturer) != ''
  `;
  for (const { manufacturer } of rows) {
    const name = toTitleCase(manufacturer);
    const mfg = await prisma.manufacturer.upsert({
      where: { name },
      create: { name },
      update: {},
    });
    await prisma.computer.updateMany({
      where: {
        manufacturer: { equals: manufacturer, mode: 'insensitive' },
        manufacturerId: null,
      },
      data: { manufacturerId: mfg.id },
    });
  }
  console.log(`Backfill complete — processed ${rows.length} distinct manufacturer values.`);
}

main()
  .catch(console.error)
  .finally(() => prisma.$disconnect());
```

### Testing Plan

1. Apply migration: `cd server && npx prisma migrate dev`.
2. Insert test Computer rows with manufacturer = `'dell'`, `'Dell'`, `''`, `null`, `'  lenovo  '`.
3. Run `npx ts-node prisma/seed-manufacturer-backfill.ts`.
4. Verify: Manufacturer table has `Dell` and `Lenovo` (two rows, normalized).
5. Verify: Computer rows with dell/Dell both point to the `Dell` Manufacturer row.
6. Verify: rows with null or empty manufacturer have `manufacturerId = NULL`.
7. Run script a second time — no errors, no duplicate Manufacturer rows.
